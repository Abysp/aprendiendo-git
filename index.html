Git Cheat Sheet

An interactive guide to the essential Git and GitHub concepts — with animated visuals to help you see how each one works.

Repository (repo)

The project folder, tracked by Git and hosted on GitHub et al.

What it does

A repository is the central storage for your project. It contains every file, folder, and the full history of every change ever made. Git tracks it locally on your machine; platforms like GitHub host a remote copy so teams can collaborate.

Considerations

  • Initialize with git init for new projects, or clone an existing one.
  • Never commit secrets (.env, API keys) — use .gitignore.
  • Keep repos focused: one project per repo is the standard.
  • README, license, and .gitignore should be set up early.

Resources

  • GitHub — Creating repositories
  • Pro Git — Getting a Git Repository
  • github/gitignore templates

Clone

Download a copy of the code to your machine.

What it does

git clone creates a full local copy of a remote repository — including all branches, commits, and history. It also sets up a connection to the remote (usually called origin) so you can push and pull later.

Considerations

  • Cloning is a one-time setup step when joining a project.
  • Use SSH keys or a personal access token for private repos.
  • Large repos can take time — consider shallow clones (--depth 1) for CI.
  • After cloning, always create a feature branch before making changes.

Resources

  • git-clone documentation
  • GitHub — Cloning a repository
  • Connecting with SSH

Branch

A safe parallel timeline to the main codebase.

What it does

A branch is an independent line of development. You create one to work on a feature or fix without affecting main. When done, you merge it back. Branches are lightweight — Git just moves a pointer.

Considerations

  • Use descriptive names: feature/dark-mode, fix/login-bug.
  • Keep branches short-lived — merge or delete them regularly.
  • Never commit directly to main in team projects.
  • Pull latest main before creating a new branch.

Resources

  • Pro Git — Branches in a Nutshell
  • Atlassian — Using branches
  • Git Flow branching model

Commit

A snapshot + message describing your changes.

What it does

A commit records a snapshot of your staged changes with a descriptive message. Each commit has a unique hash and links to its parent, forming an immutable history chain. Commits are local until you push them.

Considerations

  • Write clear messages: imperative mood, ~50 char subject line.
  • Commit often, in small logical units — not one giant commit.
  • Stage selectively with git add -p for clean history.
  • Commits are permanent in shared history — avoid force-pushing shared branches.

Resources

  • How to Write a Git Commit Message
  • git-commit documentation
  • Conventional Commits

Push / Pull

Upload to / download from the hosted repository.

What it does

git push uploads your local commits to the remote repository. git pull downloads new commits from the remote and merges them into your current branch. Together they keep local and remote in sync.

Considerations

  • Always pull before pushing to avoid rejected pushes.
  • Push feature branches early and often for backup and visibility.
  • Use git pull --rebase for a cleaner history on shared branches.
  • Never force-push to shared branches without team agreement.

Resources

  • git-push documentation
  • git-pull documentation
  • Atlassian — Syncing

Diff

The visual difference between two versions of code from two different branches.

What it does

A diff shows line-by-line changes between two states — files, commits, or branches. Added lines appear in green, removed in red. Diffs are essential for code review, debugging, and understanding what changed before merging.

Considerations

  • Review diffs before every commit and PR — catch mistakes early.
  • git diff shows unstaged changes; git diff --staged shows staged.
  • Large diffs are hard to review — keep PRs small.
  • IDEs and GitHub provide side-by-side and unified diff views.

Resources

  • git-diff documentation
  • GitHub — Reviewing PR changes
  • Tower — Understanding diffs

Merge

Combine a branch back into another branch.

What it does

Merging integrates changes from one branch into another, creating a merge commit that joins both histories. It's the standard way to bring a completed feature branch back into main after review.

Considerations

  • Merge after PR approval — not before review.
  • Prefer "Squash and merge" for clean history on feature work.
  • Update your branch with latest main before merging to reduce conflicts.
  • Merge commits preserve full history; squash merges flatten it.

Resources

  • Pro Git — Basic Branching and Merging
  • GitHub — Merge methods
  • Atlassian — git merge

Rebase

Replay your commits on top of the latest main.

What it does

Rebase moves your branch's commits to sit on top of another branch's tip, rewriting history to appear as if you started from the latest point. The result is a linear, cleaner history without merge commits.

Considerations

  • Never rebase commits that have already been pushed and shared.
  • Great for updating feature branches before opening a PR.
  • Conflicts must be resolved commit-by-commit during rebase.
  • Use git rebase -i to squash or reorder local commits.

Resources

  • Pro Git — Rebasing
  • Atlassian — git rebase
  • git-rebase documentation

Merge Conflict

Two changes touched the same lines; you choose.

What it does

A merge conflict occurs when Git can't automatically reconcile overlapping changes in the same file. Git pauses the merge and marks the conflicting sections with <<<<<<< markers. You manually edit the file, choose the correct code, then continue the merge.

Considerations

  • Don't panic — conflicts are normal in active projects.
  • Communicate with teammates when the same files are being edited.
  • Use your IDE's merge tool or VS Code's conflict resolver.
  • After resolving, always test before completing the merge.

Resources

  • GitHub — Addressing merge conflicts
  • Pro Git — Basic merge conflicts
  • Atlassian — Merge conflicts

Pull Request (PR)

"Please review and merge my branch."

What it does

A Pull Request is a request to merge your branch into another (usually main). It shows the diff, enables code review, runs CI checks, and documents the discussion before changes land in the main codebase.

Considerations

  • Write a clear PR description: what, why, and how to test.
  • Keep PRs small and focused — one concern per PR.
  • Link related issues with "Closes #123".
  • Request reviews from relevant teammates; address feedback promptly.

Resources

  • GitHub Pull Requests docs
  • GitHub Code Review
  • Atlassian — Making a pull request

Issue

A ticket: bug, feature, or task.

What it does

Issues are GitHub's task tracker. Each issue represents a bug report, feature request, or todo item. They can be labeled, assigned, linked to PRs, and organized into milestones or project boards.

Considerations

  • Use labels consistently: bug, enhancement, good first issue.
  • Write reproducible steps for bug reports.
  • Link PRs to issues so they auto-close on merge.
  • Break large features into smaller, trackable issues.
